# HTTP API Testing Coach Project Context

We are building an AI Coaching experience for HTTP API behavioral testing.

Core philosophy:

* augmentation, not automation
* visible reasoning
* intentional friction
* behavioral understanding over code generation
* manifest as both executable config and learning artifact

## Current architecture

### 1. Manifest-driven test platform

Primary artifact:

```text
api-tests.json
```

The manifest defines:

* request construction
* test data
* assertions
* expected behavior

The manifest is intended to remain:

* human-readable
* reviewable
* teachable
* executable

We discussed "manifest ergonomics" extensively:

* low noise
* readable intent
* predictable edits
* minimal hidden magic
* local reasoning
* maintainability
* coachability

### 2. Assertion library

File:

```text
assertions.js
```

Dependency-free.

Current capabilities:

* status assertions
* header assertions
* JSON assertions

Supported JSON assertions:

* exists
* equals
* contains
* oneOf
* isArray
* hasProperties
* minLength
* each

Supports:

* JSONPath-like selection
* collection iteration
* each.property assertions

Example:

```json
{
  "path": "$",
  "each": {
    "property": "status",
    "oneOf": ["active","pending","completed"]
  }
}
```

### 3. HTTP runner

File:

```text
api-test-runner.js
```

Dependency-free Node.js runner.

Supports:

* manifest loading
* $data references
* token generation
* sequential execution
* console reporting
* JSON report output
* verbose mode
* exit codes

Supported body types:

* json
* form
* text

Example:

```json
"bodyType": "form"
```

form behavior:

* URLSearchParams encoding
* content-type: application/x-www-form-urlencoded

Default:

* json

### 4. Data handling

Manifest contains embedded test data:

```json
"data": {
  "task.valid": {
    "task": {
      "id": "${randomId}",
      "title": "Buy milk",
      "status": "active"
    }
  }
}
```

Important discovery:

* runner should mirror API expectations
* manifest carries API-specific request shapes
* runner should remain generic

### 5. Reporting philosophy

Goal:

* low noise
* high readability
* quick failure comprehension

Console:

* concise by default
* verbose optional

Machine-readable report:

* always detailed

### 6. Sample API

We successfully tested a sample CRUD-style task API written in Node.js.

The platform successfully executed:

* happy-path tests
* sad-path tests
* filtering tests
* create/update operations

Important implementation discovery:

* API expected wrapped task payloads
* API expected form-encoded updates
* architecture adapted cleanly without redesign

### 7. AI Coach direction

The AI Coach should NOT simply generate manifests automatically.

The coaching flow should:

1. inspect server.js and/or API Story
2. identify behaviors
3. suggest candidate tests
4. distinguish happy/sad paths
5. review data shapes
6. review assertion choices
7. allow add/edit/remove
8. generate plausible first manifest

Key principle:

* plausible first draft
* not authoritative truth

Useful friction points:

* route review
* behavior review
* test inventory review
* assertion-choice review
* data-shape review
* enable/disable decisions

The coach should help users learn behavioral API testing while constructing executable manifests.
