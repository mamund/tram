# TRAM Project Context Handoff

## Project name

TRAM

```text
Test Runner for Assertion Manifests
```

## Core philosophy

TRAM is an AI-coaching-oriented HTTP API behavioral testing platform.

The project emphasizes:

* augmentation over automation
* behavioral understanding over scripting
* visible reasoning
* intentional friction
* executable manifests as learning artifacts
* low-noise reporting
* framework independence

The long-term direction includes an AI Coach that helps users collaboratively construct executable behavioral API tests.

## Current architecture

Current flow:

```text
server.js / API Story
        ↓
AI Coach interaction
        ↓
api-tests.json
        ↓
TRAM runner
```

### Range assertions

Implemented.

Examples:

```json
{
  "path": "$.priority",
  "range": {
    "min": 1,
    "max": 5
  }
}
```

```json
{
  "path": "$.temperature",
  "range": {
    "min": -40
  }
}
```

Rules:

```text
min optional
max optional
inclusive bounds
numeric values only
negative values supported
```

Collection example:

```json
{
  "path": "$",
  "each": {
    "property": "priority",
    "range": {
      "min": 1,
      "max": 5
    }
  }
}
```

## Body assertions

Terminology updated.

Previous:

```json
"expect": {
  "json": []
}
```

Current:

```json
"expect": {
  "body": []
}
```

Reasoning:

* assertions target response bodies
* avoids JSON-specific terminology
* prepares for additional media types

Assertions still operate on parsed JSON responses using JSONPath-like traversal.

## Data handling

Current manifests support embedded reusable data.

Example:

```json

  "task.valid": {
    "task": {
      "id": "${randomId}",
      "title": "Buy milk",

  }
```

Referenced using:

```json
"body": "$data.task.valid"
```

## Runtime token behavior

Current behavior:

```text
${randomId}
→ resolves every time encountered
```

Planned behavior:

```text
stable run-scoped variables
→ resolved once per test run
→ reused throughout execution
```

Example planned usage:

```json
"data": {
  "stableId": "${randomId}"
}
```

Then:

```json
"path": "/tasks/${stableId}"
```

Use cases:

```text
create
retrieve
update
delete
verify missing
```

## Manifest ergonomics

Key design concern:

```text
manifest ergonomics
```

Meaning:

* readable intent
* low repetition pressure
* predictable edits
* local reasoning
* maintainability
* coaching usability
* failure readability

Important architectural realization:

```text
authoring format
and
execution format
may eventually diverge
```

Current decision:

* executable JSON manifest remains primary
* no Markdown authoring layer yet
* AI Coach interaction currently acts as the human-friendly layer

## AI Coach direction

The AI Coach should:

1. inspect `server.js` and/or API Story documents
2. identify API behaviors
3. suggest candidate tests
4. distinguish happy and sad paths
5. review request/response shapes
6. review assertion choices
7. allow modification and refinement
8. generate executable manifests

Important principle:

```text
plausible first draft
not authoritative truth
```

Useful friction points:

* route review
* behavior review
* assertion-choice review
* data-shape review
* enable/disable review
