# TRAM Project Context

## Project overview

TRAM (Test Runner for Assertion Manifests) is a lightweight, dependency-free HTTP API behavioral testing platform for Node.js.

The project explores the idea that behavioral expectations themselves can become portable, executable operational artifacts.

Instead of embedding operational expectations primarily inside:

* framework-specific test code
* monitoring systems
* prose documentation
* client assumptions
* generated scaffolding

TRAM attempts to expose behavioral intent directly through explicit assertion manifests.

The core artifact is:

```text
api-tests.json
```

The manifest acts as both:

* executable configuration
* behavioral operational artifact

## Core architectural direction

The project is intentionally focused on:

* behavioral verification
* inspectable operational intent
* declarative manifests
* low-noise execution
* predictable runtime behavior
* framework independence
* augmentation-oriented workflows

TRAM is not attempting to become:

* a general-purpose testing framework
* an orchestration engine
* a plugin platform
* a schema engine
* a browser automation platform
* an embedded scripting runtime

The architecture deliberately resists introducing:

* custom scripting
* callback injection
* hidden runtime state
* imperative orchestration
* framework lock-in

## Current implementation

Current implementation includes:

* dependency-free HTTP runner
* dependency-free assertion engine
* declarative manifest specification
* runtime interpolation
* stable run-scoped variables
* collection assertions
* object-map assertions
* nested traversal assertions
* machine-readable reporting
* JSON, form, and text request support
* happy-path and sad-path testing

The current implementation is intentionally compact.

## Manifest model

The manifest defines:

* requests
* request bodies
* expected behaviors
* assertions
* shared runtime data
* runtime interpolation values

Typical structure:

```json
{
  "manifestVersion": "0.1",
  "config": {
    "baseUrl": "http://localhost:3000"
  },
  "data": {},
  "tests": []
}
```

## Runtime interpolation model

TRAM supports runtime interpolation using:

```text
${...}
```

Examples:

```json
"path": "/tasks/${data.stableId}"
```

```json
{
  "path": "$.id",
  "equals": "${data.stableId}"
}
```

Interpolation currently applies to:

* request paths
* query values
* request bodies
* expectations

Supported runtime tokens:

```text
${randomId}
${timestamp}
${uuid}
${randomEmail}
```

## Stable run-scoped variables

TRAM supports stable values initialized once per test run.

Example:

```json
"data": {
  "stableId": "${randomId}"
}
```

All later references reuse the same value during the current execution.

This allows coordinated multi-step workflows without introducing imperative scripting.

## Traversal and reference semantics

TRAM currently uses several related traversal/reference systems.

### Assertion traversal

Assertion paths operate on response bodies using JSONPath-like traversal.

Example:

```json
{
  "path": "$.status",
  "equals": "active"
}
```

### Manifest data lookup

Manifest data references retrieve reusable manifest-defined values.

Example:

```json
"body": "$data.task.valid"
```

### Runtime interpolation

Runtime interpolation inserts values during runtime request construction.

Example:

```json
"path": "/tasks/${data.stableId}"
```

## Assertion model

Current assertion support includes:

```text
exists
equals
contains
oneOf
range
isArray
hasProperties
minLength
each
eachProperty
```

## Collection assertions

TRAM supports array traversal through `each`.

Example:

```json
{
  "path": "$",
  "each": {
    "property": "status",
    "oneOf": [
      "active",
      "pending",
      "completed"
    ]
  }
}
```

This enables behavioral validation across returned collections.

## Object-map assertions

TRAM supports object-map traversal through `eachProperty`.

Example:

```json
{
  "path": "$._links",
  "eachProperty": {
    "hasProperties": [
      "href",
      "method"
    ]
  }
}
```

This allows manifests to validate affordance-oriented structures such as hypermedia link maps.

## Nested traversal support

Collection assertions and object-map assertions can be combined.

Example:

```json
{
  "path": "$",
  "each": {
    "path": "$._links",
    "eachProperty": {
      "hasProperties": [
        "href",
        "method"
      ]
    }
  }
}
```

This assertion verifies:

```text
for each record
  for each link relation
    ensure href and method exist
```

The assertion model now supports:

* collection traversal
* nested traversal
* object-map iteration
* hypermedia affordance validation

while remaining declarative and inspectable.

## Hypermedia orientation

TRAM is intentionally compatible with affordance-oriented and hypermedia-style APIs.

The project treats runtime affordances as operationally important verification targets.

Assertions may verify:

* affordance presence
* discoverability
* navigation surfaces
* state transitions
* collection invariants
* behavioral constraints

This broadens testing beyond endpoint availability alone.

## AI Coaching direction

A future AI Coaching layer is planned.

The AI Coach is intended to:

* inspect server implementations and/or API Story documents
* identify API behaviors
* propose candidate tests
* distinguish happy and sad paths
* collaboratively review assertions
* generate plausible first-pass manifests

The goal is not one-shot automation alone.

The goal is helping users understand behavioral API testing while collaboratively constructing executable manifests.

The coaching direction is explicitly augmentation-oriented rather than automation-oriented.

## Reporting philosophy

TRAM emphasizes:

* concise console output
* readable failures
* behavior visibility
* inspectable assertions
* machine-readable reports

The default reporting model intentionally avoids excessive noise.

## Explicit non-goals

TRAM currently does not attempt to support:

```text
custom scripting
plugin systems
browser automation
parallel orchestration
schema validation engines
framework-specific adapters
hidden execution hooks
```

The current architecture favors clarity and inspectability over maximal flexibility.

## Current architectural assessment

The current assertion architecture remains relatively compact while supporting:

* nested traversal
* runtime interpolation
* stable shared runtime values
* collection iteration
* object-map iteration
* hypermedia validation

without introducing:

* embedded scripting
* callback hooks
* plugin systems
* imperative orchestration

Maintaining that balance is currently considered an important design constraint.
